# جریان‌های اصلی محصول

این سند برای بازطراحی UI/UX نوشته شده است. هر صفحه یا کامپوننت باید به یکی از
این کارها کمک کند؛ اگر کمکی نمی‌کند، باید حذف، ادغام یا به بخش تنظیمات منتقل
شود.

## 1. کشف فرصت با جستجوی زنده

هدف کاربر: بدون آزمون‌وخطای زیاد، یک فیلتر دقیق بسازد و بفهمد آیا آگهی‌های
مرتبط وجود دارد یا نه.

مسیر پیشنهادی:

1. انتخاب نوع جستجو: کلیدواژه یا شماره معامله.
2. تکمیل فیلترهای مهم: نوع معامله، سامانه/زیرنوع، سازمان، دسته‌بندی، محل، مهلت
   و بازه قیمت.
3. نمایش نتیجه با ستون‌های تصمیم‌ساز: شماره، عنوان، سازمان، محل، مهلت، مبلغ و
   نوع.
4. باز کردن جزئیات هر ردیف در dialog یا پنل کناری.
5. ذخیره همان فیلتر به عنوان پایش.

خروجی موفق:

- کاربر می‌داند فیلتر چه چیزی را پوشش می‌دهد.
- ساخت پایش بدون دوباره‌کاری انجام می‌شود.
- اگر Setad پاسخ ندهد، خطا واضح و قابل retry است.

## 2. ساخت پایش زمان‌بندی‌شده

هدف کاربر: جستجوی مهم را به یک پایش تبدیل کند تا بعدا فقط تغییرها را ببیند.

مسیر پیشنهادی:

1. نام‌گذاری پایش با عنوان قابل فهم.
2. نمایش خلاصه فیلتر قبل از ذخیره.
3. انتخاب فاصله اجرا با حداقل امن سیستم.
4. انتخاب مقصدهای اعلان، یا خاموش نگه داشتن اعلان.
5. تعیین رفتار اعلان: baseline اولیه، آگهی جدید، تغییر آگهی، تغییر پیشنهاد.
6. ذخیره و رفتن به صفحه جزئیات همان پایش.

خروجی موفق:

- پایش owner مشخص دارد.
- baseline در اولین اجرای موفق ثبت می‌شود.
- بعد از baseline فقط delta ذخیره و ارسال می‌شود.

## 3. مدیریت پایش‌ها

هدف کاربر: وضعیت پایش‌ها را سریع ببیند و پایش مشکل‌دار را اصلاح کند.

مسیر پیشنهادی:

1. لیست پایش‌ها با وضعیت فعال، آخرین اجرا، اجرای بعدی، تعداد آگهی و آخرین تغییر.
2. کنترل‌های سریع: اجرای دستی، فعال/غیرفعال، ویرایش، حذف.
3. فیلتر وضعیت: فعال، غیرفعال، دارای خطا، بدون baseline.
4. ورود به جزئیات پایش برای دیدن فیلتر، رویدادها، آگهی‌های matched و run history.

خروجی موفق:

- کاربر می‌فهمد کدام پایش ارزش توجه دارد.
- حذف پایش، آگهی‌های orphan را پاک می‌کند ولی آگهی مشترک با پایش دیگر را نگه
  می‌دارد.

## 4. بررسی بروزرسانی‌ها

هدف کاربر: به جای دریافت لیست تکراری، فقط تغییرهای قابل اقدام را ببیند.

انواع رویداد:

- `baseline`: کارت اولیه برای موارد موجود در اولین اجرای موفق.
- `new_listing`: آگهی تازه که قبلا در baseline یا اجراهای قبلی نبود.
- `listing_changed`: تغییر معنی‌دار در مهلت، قیمت، عنوان، سازمان، محل یا وضعیت.
- `offer_new`: پیشنهاد جدید برای مزایده.
- `offer_changed`: تغییر مبلغ، رتبه یا وضعیت پیشنهاد.
- `removed_listing`: آگهی که دیگر در نتیجه پایش دیده نمی‌شود، اگر این رفتار فعال
  شود.

هر کارت باید حداقل داشته باشد:

- نام پایش و زمان رویداد
- شماره معامله و عنوان
- سازمان، استان/شهر، دسته‌بندی
- مهلت ارسال و مبلغ/قیمت پایه
- برای مزایده: پیشنهاددهنده، مبلغ، زمان ثبت، رتبه/وضعیت
- خلاصه تغییر با مقدار قبلی و جدید، وقتی موجود است

خروجی موفق:

- کاربر از روی کارت می‌فهمد چرا اعلان آمده است.
- ارسال Rubika همان کارت فشرده و خوانا را منتقل می‌کند.
- اجرای بدون تغییر، اعلان جدید ایجاد نمی‌کند.

## 5. تحلیل مزایده و پیشنهادها

هدف کاربر: برای آگهی‌های مزایده، تاریخچه پیشنهادها و تغییرات آن‌ها را بفهمد.

مسیر پیشنهادی:

1. در جزئیات آگهی، بخش پیشنهادها فقط برای مزایده‌های پشتیبانی‌شده نمایش داده
   شود.
2. پیشنهادها با مبلغ، پیشنهاددهنده، زمان، رتبه و وضعیت مرتب شوند.
3. تغییرهای پیشنهاد به شکل event جداگانه در inbox دیده شوند.
4. اگر history قابل دریافت نبود، پیام خطا جدا از جزئیات آگهی نمایش داده شود.

خروجی موفق:

- مزایده صرفا شبیه یک آگهی معمولی نیست.
- کاربر می‌تواند پیشنهاد جدید یا تغییر رتبه/مبلغ را سریع تشخیص دهد.

## 6. رسیدگی به خطاهای اجرا

هدف کاربر: بفهمد مشکل از Setad، فیلتر، زمان‌بندی، Rubika یا سیستم داخلی است.

مسیر پیشنهادی:

1. صفحه اجراها با وضعیت `success`، `failed` و `running`.
2. نمایش fetched، matched و changed برای هر run.
3. پیام خطای انسانی برای rate limit، timeout، rejected filters و خطای Rubika.
4. لینک مستقیم از run به پایش و رویدادهای همان run.

خروجی موفق:

- خطاها در inbox محصول گم نمی‌شوند.
- اپراتور می‌داند retry کافی است یا باید فیلتر/تنظیمات را تغییر دهد.

## 7. مدیریت مقصدهای Rubika

هدف کاربر: مقصدهای اعلان را بدون دستکاری تنظیمات سرور مدیریت کند.

مسیر پیشنهادی:

1. نمایش وضعیت bot token و default chat بدون افشای مقدار secret.
2. ساخت مقصد با نام انسانی، نوع مقصد و chat ID.
3. فعال/غیرفعال کردن مقصد.
4. ارسال پیام تست فقط برای admin.
5. انتخاب مقصدها هنگام ساخت یا ویرایش پایش.

خروجی موفق:

- مقصد غیرفعال هیچ ارسالی دریافت نمی‌کند.
- خطای ارسال روی delivery ذخیره می‌شود و در UI قابل فهم است.

## 8. مدیریت کاربران و نقش‌ها

هدف کاربر: دسترسی‌ها روشن و قابل کنترل باشند.

نقش‌ها:

- `admin`: مدیریت کاربران، مقصدها و همه پایش‌ها.
- `operator`: ساخت و مدیریت پایش‌های خودش.
- `viewer`: مشاهده read-only.

مسیر پیشنهادی:

1. لیست کاربران با نقش، وضعیت و تاریخ ایجاد/ویرایش.
2. ساخت کاربر با رمز اولیه امن.
3. ویرایش نقش، فعال/غیرفعال و reset password.
4. جلوگیری از حذف دسترسی admin برای خود کاربر فعلی.

خروجی موفق:

- هیچ صفحه عملیاتی برای viewer دکمه مخرب یا نوشتنی نشان نمی‌دهد.
- operator داده‌های کاربران دیگر را نمی‌بیند.

## 9. معیارهای قبولی بازطراحی

- اپلیکیشن باید workbench چندمسیره باشد، نه صفحه landing یا تک‌صفحه شلوغ.
- RTL و فارسی باید first-class باشند؛ متن انگلیسی فقط برای کدها، API و شناسه‌ها.
- هر مسیر باید empty، loading، error، success و permission state داشته باشد.
- داده demo باید برای بررسی همه مسیرهای بالا کافی باشد.
- صفحات نباید feature نمایشی template را نگه دارند مگر اینکه به یکی از جریان‌های
  بالا وصل شده باشد.
- تصمیم‌های مهم کاربر باید با اطلاعات کافی انجام شوند، نه با badgeهای مبهم یا
  جدول‌های بدون جزئیات.
